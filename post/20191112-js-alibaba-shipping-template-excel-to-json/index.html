JavaScript-阿里巴巴运费模板定制化需求——第一阶段：Excel To JSON

2019年11月12日
 read
   Categories: love编程

本文主要是用于记录解决「阿里巴巴国际站运费模板」输入需求，具体情况如下：

问题：

  1. 燕文运输报价单有179条数据，现在需要一个个的输入
  2. 名下有多家店铺需要同步此模板

需求：

写一个小功能实现Excel数据直接可以创建对应的运费条目

由于朋友在提交此需求之前就已经定位到能够通过修改特定域的JSON数据达到此目的，所有将解决分解为2个阶段：

功能：

  1. 第一阶段：从 EXCEL 生成对应的JSON数据，减少 Excel 一条条建项的重复工作
  2. 第二阶段：利用油猴等手段，可直接在网页导入 Excel 就可以修改对应的数据

本文即为第一阶段的主要实践过程记录，大致可以分为2部分记录：

  1. 分解页面逻辑
  2. 实现 Excel 到 JSON 的转化

第二次拆解-数据差异以及整体业务流程-20191109

数据差异点

上次已经分解出对应的 JSON 数据，需要做以下事情：

  1. 朋友提供的 Excel 数据哪些数据是有用的
  2. EXCEL 缺少 JSON 中的某些数据从哪里可以获取？缺少能否提交

第一轮对比：阿里巴巴运费设置界面与 Excel 数据对比

从以上2张图可以看出 Excel 中有用的数据为「英文名、首重 10g 价格、续重每10g 价格」，其他都是无用的

第二轮对比：Excel 与 JSON结构对比

这一轮对比应该可以判断 Excel 中仅有 「defineWeights」、「shippingCountry」两个数据点，其他的数据点都无，所以需要逐减少项目来测试是否影响整体的提交

所以最基础的需求就是以 Excel 的数据生成以上的 JSON 结构

反过头来看看能在 最简单的 JSON 结构上增加哪些内容呢，在反复的过程中发现可以增加以下字段：

  • standardShipCountry：标准邮费国家，系统存储在DOM结构中，在设置完毕之后自动生成

程序逻辑走向-一个错误思考分支走向

上一章节通过对比相关数据提出一个 构成全新的最简JSON结构填充到textarea，但是在开始之初不知道是哪种想法导致想要 模拟真实逻辑找到真正的数据存储点修改对应的数据所以有如下动作：

分解程序整体业务逻辑：
  1. 点击某个模板：
  2. 展示模板下所有的物流页面（分解「tbody」的数据可以看到已提交的数据内容如上文分析的「第一次拆解-定位数据位置与结构-20191105-JSON 数据拆解 」）
  3. 点击某个物流编辑跳出弹窗编辑相关数据（注意此次为同一个页面）
  4. 点击模板的保存界面，将所有的数据再次提交至于服务器

主要的业务逻辑是：

  2. 利用读取数据拼接成相关的代码

以下就是第一步的相关代码

$('#shipFile').change(function(e){
    var files=e.target.files;
    var fileReader=new FileReader();
    //读取Excel文件
    fileReader.onload=function(ev){
        try{
            var data = ev.target.result;
            //以二进制读取整个表格
            var workbook=XLSX.read(data,{type:"binary"});
            //物流数据数组
            var shipDataArray=[]; 
            var fromTo = '';

            //遍历每张表读取
            for (var sheet in workbook.Sheets) {
                if (workbook.Sheets.hasOwnProperty(sheet)) {
                    fromTo = workbook.Sheets[sheet]['!ref'];
                    shipDataArray = shipDataArray.concat(XLSX.utils.sheet_to_formulae(workbook.Sheets[sheet]));
                }
            }
        }catch(e){
            console.log('文件类型不正确');
            return;
        }
    }
    // 以二进制方式打开文件
    fileReader.readAsBinaryString(files[0]); 
}

第二次编码-20191112

本次主要是优化相关体验，在第一次编码结束之后提出以下问题：

  1. 第一次编码未将相同运费的国家归为一个「defines」,导致传输的数据远比官网大而且加载过慢：必须解决
  2. 国家列表需要手工输入，不合实际请款
  3. 初始重量、续重单位均是固定，不满足后续可扩展性能

分析相关情况之后提出以下的功能调整思路：

  1. 建立 Map 存储点，key为运费具体情况，value为国家字符串===>第1个问题
  2. 完成的列出每个物流对应的国家列表，然后用下拉框选择即可
  3. 将初始重量等设置为变量
